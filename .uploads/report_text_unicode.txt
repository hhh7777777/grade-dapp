1、需求分析
1.1 课题背景
　　随着高校教学管理信息化程度不断提高，成绩录入、成绩查询和成绩修改等业务通常依赖中心化数据库与后台管理权限。中心化系统虽然便于集中维护，但也存在数据修改痕迹不透明、管理员误操作难追溯、不同系统之间校验成本较高等问题。区块链具有链上交易可追踪、合约逻辑公开执行、数据记录难以被单点篡改等特点，将其引入成绩管理场景，可以作为课程中理解 DApp 架构、智能合约权限控制、前后端与链交互流程的典型案例。
　　本项目选择课程设计“选题一：基于以太坊的链上学生成绩管理 DApp 开发”。按照课程考查要求，项目需要提交项目说明书和项目源代码，材料以“学号+姓名+选题”的方式命名，并且同一选题下不同学生内容不能雷同。因此，本报告围绕 grade-dapp 项目的真实代码结构撰写，突出本项目采用的合约结构、服务接口、前端交互与测试用例，而不是照搬通用模板。
1.2 设计目标
　　（1）完成一套可以本地运行和答辩演示的链上学生成绩管理 DApp，覆盖合约部署、后端启动、前端启动和 MetaMask 连接本地链等完整流程。
　　（2）实现严格的权限隔离：合约部署者为唯一管理员，只有管理员地址能够录入或修改成绩，普通用户只能查询成绩。
　　（3）实现成绩数据链上存储：以学号为索引保存学生姓名、成绩、备注、更新时间、更新人地址等字段，查询不存在学号时返回合法提示。
　　（4）实现前后端与链路联调：后端通过 ethers.js 读取部署文件中的合约地址和 ABI，前端通过 API 查询成绩并通过 MetaMask 发起管理员写交易。
　　（5）形成规范课程设计报告：报告包含需求分析、总体设计、详细设计、测试分析、结论心得、参考文献，以及程序框图或流程图。

1.4 非功能需求与约束条件
　　安全性方面，系统必须保证写入权限只能由管理员地址执行，非管理员调用合约写入函数时应被拒绝；输入校验方面，学号和姓名不能为空，成绩必须是 0 到 100 之间的有效数字，避免无效数据永久写入链上；稳定性方面，后端应在本地链未启动或部署文件缺失时返回清晰错误，而不是直接崩溃；易用性方面，前端需要在查询、提交、等待确认和失败等状态下给出明确提示；可演示性方面，项目不追求复杂生产部署，而是保证本地测试链、后端、前端和钱包四者能够形成闭环。
2、总体设计
2.1 系统总体架构
　　系统采用典型 DApp 分层结构。用户在浏览器中访问 React 前端页面，普通用户只需要输入学号即可查询成绩；管理员需要通过 MetaMask 连接本地Ganache 网络，系统会读取当前钱包地址并与合约管理员地址进行比较。查询操作通过后端 API 完成，写入操作由前端直接创建合约实例并通过 MetaMask 进行签名，从而保证链上交易的签名地址与浏览器当前钱包一致。后端主要负责链路状态检查、只读查询、事件查询、交易状态查询和统一响应封装。

2.3 程序流程设计
　　程序流程分为公开查询流程和管理员成绩上链流程。公开查询流程不要求连接钱包，用户输入学号后，前端调用后端 API，后端再通过 ethers.js 调用合约 getGrade 函数，最后根据 exists 字段决定展示成绩详情或“暂无记录”。管理员上链流程要求浏览器存在 MetaMask、当前网络为 Hardhat 本地链、当前钱包地址与合约管理员地址一致，并且录入表单满足基本校验。校验通过后，前端通过钱包签名调用 setGrade，等待 receipt 后展示交易 Hash、区块号和 Gas Used。

3、详细设计
3.1 智能合约实体设计
　　智能合约核心实体包括管理员地址、成绩记录、成绩映射和成绩更新事件。管理员地址 admin 使用 immutable 修饰，部署后不可再修改，能够减少运行成本并防止管理员被后续代码意外覆盖。成绩记录 GradeRecord 不是只保存分数，而是同时保存学号、姓名、分数、备注、更新时间、更新人地址和 exists 标记，便于查询页面展示完整的链上成绩信息。

字段/结构	类型	设计说明
admin	address immutable	部署者地址自动写入，作为唯一管理员。
GradeRecord.studentId	string	保存原始学号，便于查询结果回显。
GradeRecord.studentName	string	保存学生姓名。
GradeRecord.score	uint256	保存成绩，合约限制 score <= 100。
GradeRecord.remark	string	保存考试类型、补考、重修等备注信息。
GradeRecord.updatedAt	uint256	保存 block.timestamp，记录最近更新时间。

grades	mapping(bytes32 => GradeRecord)	以 keccak256(studentId) 作为键定位成绩记录。

3.3 后端接口设计
　　后端采用 Express 搭建 REST API，入口文件 app.js 负责加载环境变量、启用 CORS、解析 JSON、挂载 /api 路由、处理 404 和全局异常。路由层只负责路径分发，控制器层负责参数校验和响应组织，服务层负责和区块链节点、合约实例进行交互。这种分层方式让接口设计与链上调用解耦，便于在报告中清楚说明每个模块的职责。

3.4 前端页面设计
　　前端采用单页结构，核心页面为 Home.jsx。页面顶部展示系统运行状态、当前钱包、当前角色和目标链信息；查询模块允许任意用户输入学号并查看结果；管理员模块展示当前钱包、管理员地址、当前角色、网络校验结果，并提供学号、姓名、成绩、备注输入框。提交按钮会根据钱包连接、网络是否正确、是否为管理员钱包以及表单是否完整自动启用或禁用。

3.5 权限控制与异常处理设计
　　权限控制由链上和前端两层共同实现。链上以 onlyAdmin 作为最终约束，任何非管理员地址直接调用 setGrade 都会回滚；前端以钱包地址与管理员地址比较作为交互层约束，普通用户看到自己的角色为普通用户，提交按钮无法使用。异常处理方面，前端在未安装 MetaMask、未连接钱包、网络错误、非管理员、查询失败、交易失败等情况下给出中文提示；后端将成功响应统一为 success、message、data，将失败响应统一为 success=false、message 和 details，便于前端稳定解析。

4.3 MetaMask 本地链配置
　　为了演示真实的钱包身份切换，需要在 MetaMask 中添加 Hardhat 本地网络：RPC URL 为 http://127.0.0.1:8545，Chain ID 为 31337，货币符号为 ETH。部署合约使用的第一个 Hardhat 测试账户作为管理员账号；第二个或第三个测试账户作为普通用户账号。当前钱包地址等于合约管理员地址时，页面允许进入管理员录入模块；当前钱包不是管理员时，页面只允许成绩查询。
　　
4.4 部署产物与后端连接
　　部署脚本 deploy.js 在合约部署成功后读取 GradeManager 的 artifact，并将 network、address、abi 和 deployedAt 写入 deployments/localhost/GradeManager.json。后端启动后通过 contract.js 读取该文件，再使用 ethers.Contract 创建只读合约实例或绑定管理员钱包的写入合约实例。该设计避免在后端代码中硬编码合约地址，提高了本地重新部署后的可维护性。

　　通过本次课程设计，我完成了一个从智能合约到前端页面的完整 DApp 项目。项目虽然规模不大，但覆盖了以太坊 DApp 开发的关键链路：合约设计、Hardhat 本地链、合约部署、ABI 与地址导出、后端接口、React 页面、MetaMask 钱包连接、交易签名、交易回执和自动化测试。与单纯编写一个合约相比，本项目更强调“链上合约”和“链下服务/前端”的协同，能够体现分布式应用开发课程的综合性要求。
　　在调试过程中，主要困难集中在三个方面：第一，链上写入必须由管理员地址签名，如果后端或前端使用的账户不一致，就会出现权限错误；第二，合约重新部署后地址会变化，前后端必须读取最新 deployments/localhost/GradeManager.json，否则会调用到旧地址；第三，MetaMask 网络需要手动切换到 Hardhat 本地链，Chain ID 与 RPC 配置错误会导致交易无法发送。针对这些问题，系统通过健康检查接口、管理员地址接口、网络校验和页面提示进行了处理。
　　本项目仍有可扩展空间。例如，可以增加批量导入成绩、成绩修改历史列表、管理员多签审批、学生隐私保护、后端数据库缓存和部署到测试网等功能。由于课程设计重点是本地以太坊 DApp 的完整闭环，本次实现优先保证核心功能正确、结构清晰、能够演示和便于说明。总体来看，本次设计使我进一步理解了智能合约权限控制、链上数据结构设计、ethers.js 合约调用、前后端分层和交易状态展示等知识，也为后续完成更复杂的区块链系统打下了基础。

课程论文格式要求：

　　　一、一般性要求
　　　（一）课程论文包括各门课程的大作业、课程考试论文、社会调查（实践）报告等；
　　　（二）字数要求：按各门课程老师的要求；若没有具体要求，则一般不少于3000字；
　　　（三）论文资料按以下顺序排列：
　　　1.封面；
　　　2.中文摘要（包括关键词）。外文论文的需要外文摘要（包括关键词）；
　　　3.正文；
　　　4.参考文献；
　　　5.课程论文评分标准及成绩评定表
　　　二、课程论文的打印与装订
　　　论文一律采用A4纸，左侧装订。
　　　（一）页面设置
　　　论文要求纵向打印，页边距的要求为：
　　　上（T）：2.5 cm
　　　下（B）：2.5 cm
　　　左（L）：2 cm
　　　右（R）：2 cm
　　　装订线（T）：0.5 cm
　　　装订线位置（T）：左
　　　其余采取系统默认设置。
　　　（二）排式与用字
　　　文字图形一律从左至右横写横排。
　　　文字一律通栏编辑。
　　　论文中文采用宋体，除特殊需要，一般不使用繁体字；外文采用Times New Roman字体，字迹清楚整齐。
　　　（三）段落设置
　　　采用多倍行距，行距设置值为1.25。
　　　其余采取系统默认设置。
　　　（四）页眉、页脚设置
　　　页眉为论文题目（不包括副题目）居中，采用五号宋体字。
　　　页脚需设置页码，页码自正文第一页起至参考文献结束止，页码采用五号黑体字，加粗，居中放置，格式如：1，2，3……页。
　　　三、论文撰写的规范格式
　　　（一）封面
　　　1.封面格式见参考模板。
　　　2.中文摘要
　　　摘要：“摘要”两字在第一行居中位置，使用小二号黑体字，加粗。内容使用小四号宋体字。起行空两格，回行顶格。中文摘要一般不超过论文字数的5%。
　　　关键词：中文摘要后换行，首行空两格，“关键词”三字加粗，后加冒号与关键词隔开，各关键词之间用分号隔开。关键词一般在3—8个之间，用小四号宋体字。
　　　3.外文摘要：“Abstract”英文单词在第一行居中位置，使用小二号黑体字，加粗。内容使用小四号Times New Roman字体。起行空两格，回行顶格。外文摘要应与中文摘要相对应。
　　　关键词：外文摘要后换行，首行空两格，“Key words”英文单词加粗，后加冒号与关键词隔开，各关键词之间用分号隔开。外文关键词应与中文关键词相对应，用小四号Times New Roman字体。
　　　（二）正文
　　　正文中文一般使用小四号宋体字，英文一般用小四号Times New Roman字体，重点文句加粗。
1.标题层次。
　　　通行的标题层次一般使用以下2种格式（实际使用可能有些小异）
　　　第一种	　　　第二种
　　　一、……	　　　1.……
　　　（一）……	　　　1.1……
　　　1.……	　　　1.1.1……
　　　（1）……	　　　
　　　	　　　
　　　	　　　
　　　各层标题均单独占行。各层级标题序数均应左对齐空2格放置，后接标题内容。一级标题使用四号宋体字，加粗；其余使用小四号宋体，加粗，末尾不加标点。
　　　各专业根据实际情况，选定以上其中的一种格式，但所采用的格式必须符合表格规定并前后统一，不得混杂使用。同一专业本科生的毕业论文标题格式应由所在学院作统一要求。毕业论文的全部标题层次应整齐清晰，相同的层次应采用统一的表示体例，正文中各级标题下的内容应同各自的标题对应，不应有与标题无关的内容。
　　　2.量和单位。各种计量单位一律采用国家标准GB3100—GB3102-93。非物理量的单位可用汉字与符号构成组合形式的单位。
　　　3.标点符号。标点符号应按照国家新闻出版署公布的“标点符号使用方法”的统一规定正确使用，忌误用和含糊混乱。
　　　4.外文字母。外文字母采用我国规定和国际通用的有关标准写法。要分清正斜体、大小写和上下脚码。
　　　5.名词、名称。科学技术名词术语采用全国自然科学技术名词审定委员会公布的规范词或国家标准、部标准中规定的名称，尚未统一规定或叫法有争议的名称术语，可采用惯用的名称。
　　　6.数字。文中的数字，除部分结构层次序数和词、词组、惯用语、缩略语、具有修辞色彩语句中作为词素的数字必须使用汉字外，应当使用阿拉伯数码，同一文中，数字表示方法应前后一致。
　　　7.公式。公式一般居中放置；有编号的公式顶格放置，编号需加圆括号标在公式右边，公式与编号之间不加虚线。
　　　公式下有说明时，应在顶格处标明“注：”。
　　　较长公式的转行应在加、减、乘、除等符号处。
　　　8.表格和插图。
　　　（1）表格。每个表格应有自己的表序和表题。表内内容应对齐，表内数字、文字连续重复时不可使用“同上”等字样或符号代替。表内有整段文字时，起行处空一格，回行顶格，最后不用标点符号。
　　　（2）插图。每幅图应有自己的图序和图题。一般要求采用计算机制图。
　　　文中图表需在表的上方、图的下方排印表号、表名、表注或图号、图名、图注，并在图表的右下角表明来源。例：
　　　图1.1   图1.2   图2.1   图2.2……（标在图下方）
　　　表1.1   表1.2   表2.1   表2.2……（标在表上方）
　　　（三）注释
　　　注释采用页末注（将注文放在加注页的页脚）或篇末注（将全部注文集中在文章末尾），不可行中加注。注释编号选用带圈阿拉伯数字，注文使用小五号宋体字。以下为引用各类文献注释格式：
　　　专著：注释编号.作者.专著.书名[M].出版社,出版年.起止页码
　　　期刊：注释编号.作者.期刊.题名[J].刊名,出版年(卷、期):起止页码
　　　论文集：注释编号.作者.论文名称: 论文集名[C].出版地:出版社,出版年度.起止页码
　　　学位论文：注释编号.作者.题名[D].保存地点:保存单位,写作年度.
　　　专利文献：注释编号.专利所有者.题名[P].专利国别:专利号, 出版日期
　　　光盘：注释编号.责任者.电子文献题名[电子文献及载体类型标识]，出版年(光盘序号) 
　　　互联网：注释编号.责任者.文献题名.电子文献网址.访问时间（年-月-日）
　　　文献作者3名以内的全部列出；3名以上则列出前3名，后加“等”(英文加“etc"”)
　　　（四）参考文献
　　　参考文献一律放在文后，其书写格式应根据GB3469-83《文献类型与文献载体代码》规定，以单字母方式标识：M专著，C论文集，N报纸文章，J期刊文章，D学位论文，R研究报告，S标准，P专利；对于专著、论文集中的析出文献采用单字母“A”标识，其他未说明的文献类型，采用单字母“Z”标识。
　　　“参考文献”四字居中放置，使用小二号黑体字，加粗。
　　　内容使用小四号宋体字，居左，空两格放置。具体结构格式与标注方法同注释中交代引文出处的注文格。
　　　书籍格式：[编号]作者.书名（版本）[M].出版地：出版社出，版时间.
　　　期刊格式：[编号]作者.文章题目.期刊名[J].年份，卷号（期数）：页码. 
　　　网络资源：[编号]作者.文章题目，发表日期；网络资源路径.
　　　四、纪律要求
　　　1.不得抄袭，如发现雷同部分超过30%的，均按不及格处理。
　　　2.不足3000字的，成绩按缺考处理。
